# Veriskein 功能 / 性能 / 创新性对比分析

> 本文是参赛作品提交材料之一，对应"功能挑战"赛道要求的"项目测试结果的功能/性能/创新性等的分析（包括与类似项目的对比分析）"。
>
> 文档许可：本文档遵循 CC-BY-SA 4.0。文中对第三方项目能力的描述基于其公开文档与公开资料（截至 2026 年），仅用于客观技术对比。

## 1. 背景与对标对象选择

赛题要求构建"基于 eBPF 的非侵入式多智能体异常监测系统"，并在参考资料中点名 **Pixie** 与 **Cilium** 两个 eBPF 生态项目。为了让对比覆盖"可观测性"与"运行时安全"两条主流技术路线，本文在赛题点名的两个项目之外，再补充两个业界最具代表性的 eBPF 运行时安全项目 **Falco** 与 **Tetragon**：

| 对标项目 | 定位 | 选择理由 |
|---|---|---|
| **Pixie** | Kubernetes eBPF 可观测性 | 赛题点名；与本项目同样用 uprobe Hook OpenSSL 截获 TLS 明文，是"应用层数据捕获"维度最直接的对照 |
| **Cilium** | Kubernetes eBPF 网络/安全/可观测性 | 赛题点名；其 Hubble 提供 L3–L7 网络可观测性，是"网络维度捕获"的对照 |
| **Falco** | eBPF 运行时安全检测（CNCF 毕业项目） | 业界最成熟的"系统调用异常检测"基线，与本项目的安全 detector 路线直接可比 |
| **Tetragon** | eBPF 运行时安全检测 + 内核态强制（Cilium 子项目） | 代表 eBPF "检测 + 阻断"的最强工程化形态，用于界定本项目的能力边界 |

**一句话定位差异**：上述四个项目都是面向**容器 / 微服务 / 主机**的通用可观测性或运行时安全工具；而 Veriskein 面向一个它们均未专门解决的新问题——**端侧多智能体（AI Agent）运行时的异常监测，尤其是"模型输出（Prompt）→ 底层系统调用"的跨层因果归因**。

## 2. 功能对比

### 2.1 能力矩阵

| 能力维度 | Pixie | Cilium / Hubble | Falco | Tetragon | **Veriskein** |
|---|---|---|---|---|---|
| 进程事件捕获（exec/fork/exit） | ✓ | 部分 | ✓ | ✓ | ✓ |
| 文件操作捕获（open/unlink/rename…） | 部分 | ✗ | ✓ | ✓ | ✓ |
| 网络连接捕获（IP/端口） | ✓ | ✓（强项） | ✓ | ✓ | ✓ |
| 真实文件路径归一化（符号链接/相对路径/dirfd） | 部分 | ✗ | ✓ | ✓ | ✓ |
| TLS 明文截获（uprobe OpenSSL） | ✓ | ✗ | ✗ | ✗ | ✓ |
| 提取 Prompt / Response 等应用层交互语义 | 协议级（HTTP/gRPC…） | ✗ | ✗ | ✗ | ✓（面向 LLM 交互） |
| 安全异常检测（非预期 Shell / 敏感文件 / 越界删除） | ✗ | 有限网络策略 | ✓（规则引擎） | ✓（策略） | ✓ |
| 资源/逻辑异常（高频 API + 重复 Prompt + 无意义 IO 综合判死循环） | ✗ | ✗ | 有限 | 有限 | ✓ |
| **多智能体 session/role 语义建模** | ✗ | ✗ | ✗ | ✗ | ✓ |
| **跨层因果归因：Prompt → 高危 syscall** | ✗ | ✗ | ✗ | ✗ | ✓（核心） |
| **跨 Agent 提示词注入因果链（CAPI）** | ✗ | ✗ | ✗ | ✗ | ✓（核心） |
| 可解释证据链（excerpt→prompt→syscall） | ✗ | ✗ | 规则命中上下文 | 事件上下文 | ✓ |
| 诚实降级（能力不可用时显式标注 visibility/fallback） | ✗ | ✗ | 有限 | 有限 | ✓ |
| 内核态阻断/强制（kill/block syscall） | ✗ | ✓（网络） | ✗（仅检测） | ✓（强项） | ✗（仅检测，设计取舍） |
| 部署形态 | 依赖 Kubernetes | 依赖 Kubernetes | 主机/容器 | 主机/容器（偏 K8s） | **端侧单机，无需 K8s** |

### 2.2 关键差异解读

- **Pixie** 与本项目在"用 uprobe Hook `SSL_read`/`SSL_write` 截获 OpenSSL 明文"这一点上技术同源，但目的根本不同：Pixie 把明文用于**协议级可观测性**（HTTP/gRPC/MySQL 等请求追踪与服务地图），不做异常检测，也不理解 LLM 的 Prompt 语义，更不把它与后续系统调用做因果关联；且 Pixie 强依赖 Kubernetes，本地数据仅约 60 分钟保留，定位是"集群内快速排障"。本项目把明文进一步**抽象为 `PromptObject`，并作为跨层因果链的上游证据**。

- **Cilium / Hubble** 的强项是 L3–L7 **网络**可观测性与网络策略，能回答"哪个 Pod 连了哪个服务、什么状态码、是否被放行"，但它不采集文件/进程级安全语义，不截获 TLS 明文，也不面向多智能体场景。

- **Falco** 是"系统调用规则引擎"的标杆：它能检测非预期 Shell、敏感文件访问等，与本项目的**基础安全 detector 高度可比**。但 Falco 是**纯检测、规则模式匹配**，规则作用域是单条/单进程 syscall，**无法跨越"语义鸿沟"**——它看不到 TLS 明文里的 Prompt，也无法建立"某条 Prompt 诱导了这次高危操作"的因果链，更没有多智能体 session/agent 的一等建模。

- **Tetragon** 代表 eBPF 运行时安全的工程化上限：基于 kprobe/tracepoint/**LSM hook**，可在内核态**直接阻断 syscall 或 SIGKILL 进程**，开销可低至亚 1%。它在"强制"维度强于本项目，但同样停留在**内核行为层**，不理解应用层 Prompt 语义，不做跨层因果归因。

> 结论：在"系统调用层捕获 + 安全检测"这一交集上，Veriskein 与 Falco/Tetragon 同台；在"TLS 明文捕获"上与 Pixie 同源；但**"应用层 Prompt 语义 ←→ 内核层 syscall 行为"的跨层因果归因、以及多智能体并发场景下的根因定位，是上述四个项目都不具备、而正是赛题最看重（功能实现与异常回溯占 40%）的能力**。

## 3. 性能对比

赛题要求"数据捕获功能的性能损耗不大于 5%"。本项目用 `tests/perf/run.sh` 做了**同一负载、仅切换捕获模式**的可复现基准（方法学见 `tests/perf/README.md`）。各模式用**真实守护进程开关**隔离：`kernel-only` 用 `--disable-tls --disable-content-io` 真正只挂 proc/fs/net；`full` 让负载在 `claude` agent 包装下真实读写 stdio，从而**真正命中内容捕获白名单路径**（由各模式 `*.daemon.log` 的事件/丢弃计数佐证：`full` 事件量约为纯内核模式的 4–5 倍，且是唯一出现捕获环丢弃的模式）。测量采用**交织取样 + 多趟中位数**（每趟先 baseline 再依次各模式），消除块状排序带来的缓存/睿频漂移偏差。实测（Linux 6.6 / WSL2，8 vCPU，release，代表性编译+TLS+stdio 负载，7 趟弃 1）：

| 模式 | 捕获内容 | 实测开销 | 赛题目标 |
|---|---|---|---|
| kernel-only | 进程/文件/网络系统调用 | **+2.18%** | ≤5% ✓（严格门控） |
| kernel+tls | + OpenSSL uprobe 明文截获 | **+2.27%** | ≤5% ✓（严格门控） |
| full | + stdio/MCP 内容采样（真实命中白名单） | **+4.04%** | ≤5% ✓（信息项） |

守护进程常驻内存 ~180–210 MiB（< 200 MiB 目标量级）。数据来源：`artifacts/perf-real/`。

横向对照（均来自各项目公开资料，量纲/场景不完全一致，仅供量级参考）：

| 项目 | 公开开销量级 | 说明 |
|---|---|---|
| Pixie | 集群 CPU < 5%（多数 < 2%） | 全集群可观测性聚合口径 |
| Falco | 约 1–5% CPU（0.40 起默认现代 eBPF 驱动） | 规则引擎 + 用户态评估 |
| Tetragon | 亚 1% | 内核态过滤，上下文切换少 |
| **Veriskein** | **数据捕获 ~2%，含内容捕获 ~4%** | 同负载、仅切捕获模式的隔离测量 |

**诚实说明**：核心数据捕获损耗（内核 syscall 捕获 + TLS 明文捕获）稳定地低于 5% 目标；含 stdio/MCP **内容捕获**的全开模式本次约 +4%，但其开销随被捕获内容吞吐量上升，在持续大量流式输出时可能逼近或超过 5%——这是"逐字节内容数据面"的固有特性，生产环境通过内容捕获限速与按角色白名单加以约束。此外，本基准的单次负载噪声约 ±6%（与捕获信号同量级），因此各百分比在重复运行间会浮动（kernel-only/kernel+tls 落在 ~0–2.3%，full 落在 ~4–6.5%）；可复现的稳健结论是"**syscall+TLS 捕获损耗小且 ≤5%，内容捕获真实且受吞吐量约束**"。我们如实给出范围、边界与方法学局限，而非只报一个最优数字。后续方向：**内核态 cgroup/pid 预过滤** + 内容捕获自适应限速。

## 4. 创新性分析

### 4.1 核心创新：跨层语义因果链（Prompt → Syscall）

现有工具按"层"切分：Pixie 看应用协议层、Hubble 看网络层、Falco/Tetragon 看内核 syscall 层。**没有任何一个把"应用层模型输出"与"内核层系统行为"用因果关系连起来**。本项目的核心贡献是构建了一条**受控、可解释的跨层因果链**：

```
工作区文件摘录(SourceArtifact) → 下游 Agent 的 TLS Prompt(PromptObject) → 下游高危 syscall
                         └──────────── EvidenceChain + CAPI 评分 ────────────┘
```

它直接回应赛题进阶要求与评审要点中点名的"突破内核态与用户态的'语义鸿沟'……精准回溯'多智能体交互中特定 Prompt 引发底层高危操作'的因果链路"。

### 4.2 多智能体一等建模与跨 Agent 归因

- **session / agent / role 一等建模**：在进程交织的并发环境中，通过 seeds + env + argv + connect 证据做 session election，做角色分类（如 `root_agent`/`sub_agent`/`mcp_server`），并保证角色升级时 `agent_id` 稳定、`lineage_id` 跨守护化churn 稳定。
- **跨 Agent 提示词注入（CAPI）**：Agent A 写入的内容被 Agent B 读取并注入其 Prompt、进而触发高危操作时，系统用 MinHash/归一化签名 + 显式传播事实（先于文本相似度）建立**跨 session 的因果链**，输出 `chain_id`、`causal_score` 与 `excerpt_match/prompt_ref/syscall` 多类证据。这是通用安全工具完全没有的能力维度。

### 4.3 工程层面的"诚实降级"

告警内置 `fallback.mode` 与 `visibility`（full/partial/unsupported/unavailable）。当 TLS 不可见或 ringbuf 背压导致内容截断时，系统**主动降级并显式标注**，而非给出无依据的高置信结论（例如 CAPI 路径在降级时不会输出 `critical`）。这种"可解释 + 不过度宣称"的工程态度，对应评审要点的"可重现性、工程性"。

### 4.4 与往届/同类作品的差异定位

本项目并非在某个现有作品上做增量，而是**复用成熟 eBPF 基础设施（libbpf、OpenSSL uprobe 这类业界通行技术路线，与 Pixie 同源）来解决一个新问题域**。相对于通用工具：

- 比 Falco/Tetragon 多了**应用层 Prompt 语义**与**跨层因果归因**；
- 比 Pixie/Hubble 多了**面向多智能体的安全异常检测与根因定位**；
- 不依赖 Kubernetes，面向**端侧单机**的智能体协同场景。

## 5. 局限与边界（诚实声明）

为符合赛事诚信与可复现要求，明确列出本项目当前不及对标项目之处与边界：

1. **不做内核态阻断**：定位为非侵入式"检测 + 归因"，不像 Tetragon 那样 kill/block；这是有意取舍（降低对智能体正常运行的干扰风险）。
2. **TLS 兼容性受限**：仅支持**动态链接的 OpenSSL**（`libssl.so.1.1/3`），不支持静态链接、BoringSSL、Go `crypto/tls`、GnuTLS 与 HTTP/2；TLS uprobe 目前仅 **x86_64**。Pixie 在协议与链接方式覆盖上更广。
3. **成熟度与规模**：Pixie/Cilium/Falco/Tetragon 是经过大规模生产验证的 CNCF 级项目，拥有完善生态、UI 与多 SIEM 集成；本项目是面向赛题的研究型原型，单机、无可视化 UI（仅 daemon + CLI + IPC）。
4. **性能边界**：在系统调用极端密集的突发场景下，全开模式开销可能超过 5%（见 §3）。

## 6. 结论

- **功能**：在赛题最看重的"跨层因果归因 + 多智能体根因定位"上，Veriskein 提供了 Pixie/Cilium/Falco/Tetragon 均不具备的能力；在系统调用捕获与基础安全检测上与 Falco 同级，在 TLS 明文捕获上与 Pixie 同源。
- **性能**：真实负载下核心数据捕获（syscall + TLS）开销约 +2%，稳定满足赛题 ≤5% 要求；含 stdio/MCP 内容捕获的全开模式约 +4%（随内容吞吐量浮动），量级与业界 eBPF 工具相当。
- **创新性**：核心贡献是"Prompt → Syscall"的受控可解释跨层因果链与跨 Agent 提示词注入归因，填补了通用 eBPF 工具在"AI 智能体运行时语义安全"这一新问题域的空白。
