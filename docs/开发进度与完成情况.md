## 0. 阶段划分说明（避免编号误解）

- 本项目开发阶段共 **7 个：Phase 0 ~ Phase 6**，无 Phase 7 及以后。
- 注意区分"开发阶段编号（Phase 0–6）"与文档内部的"章节编号"，二者含义不同，请勿混淆。

## 3. 性能达标情况（PPT 性能页，重点亮点）

方法：同一代表性负载，仅切换挂载的 BPF 程序（真实守护进程开关隔离），
交织取样 + 多趟中位数，公平对比"纯捕获开销"。`full` 模式让负载在 agent 包装下
真实读写 stdio，**真正命中内容捕获白名单路径**（事件量约为纯内核模式 4–5 倍）。

| 模式        | 捕获开销   | 门控                       | 结论        |
| ----------- | ---------- | -------------------------- | ----------- |
| kernel-only | **+2.18%** | ≤ 5%（严格）               | ✅          |
| kernel+tls  | **+2.27%** | ≤ 5%（严格）               | ✅          |
| full        | **+4.04%** | 信息项（随内容吞吐量变化） | ✅ 本次达标 |

- daemon 常驻内存约 **180–210 MiB**（< 200 MiB 目标量级），开销随挂载层数单调递增，物理自洽。
- 诚实标注：单次负载噪声约 ±6%，与捕获信号同量级，各百分比在重复运行间会浮动
  （kernel-only/kernel+tls ~0–2.3%，full ~4–6.5%）；稳健结论为"syscall+TLS 捕获
  ≤5%，内容捕获真实且受吞吐量约束"。内容捕获在持续大流量下可能逼近/超过 5%，
  生产环境以限速+按角色白名单约束，未来方向为内核态 cgroup/PID 预过滤。
- 数据来源：`artifacts/perf-real/`（含 `report.{json,md}` 与各模式 `*.daemon.log` 真实事件/丢弃计数）。
