## 1. 项目概述与赛题对应

### 1.1 项目定位

Veriskein 是一个面向**端侧多智能体（AI Agent）运行时**的非侵入式异常监测系统。它以操作系统内核视角，利用 eBPF（tracepoints + uprobes）在**不修改内核源码、不重编译目标智能体程序**的前提下采集进程、文件、网络与 TLS 明文等多层数据，在用户态完成会话/角色归因、内容重组、跨层因果关联与异常检测，最终输出带完整上下文与可解释证据链的统一 JSON 告警。

核心目标是**跨越"语义鸿沟"**：把"模型输出（Prompt/Response）"这一应用层语义与"底层系统调用行为"用因果关系连接起来，回答"是哪一条 Prompt 诱发了这次高危操作"，并在多智能体并发、进程交织的环境中精确定位异常根因。

### 1.2 与赛题任务的对应

| 赛题任务 | 要求 | Veriskein 实现 |
|---|---|---|
| **多层级数据捕获 · 基础** | eBPF 监控 execve/fork、open/unlink、网络连接 IP/端口 | `bpf/proc.bpf.c`、`bpf/fs.bpf.c`、`bpf/net.bpf.c`（tracepoint，`sys_exit_*` 保留返回值） |
| **多层级数据捕获 · 进阶** | uprobes Hook OpenSSL 非侵入式截获 HTTPS 明文 | `bpf/tls_uprobe.bpf.c` 全局 attach `SSL_read{,_ex}` / `SSL_write{,_ex}`，捕获明文 → `PromptObject` |
| **异常监测 · 基础（资源/逻辑）** | 高频 API + 重复 Prompt + 无意义 IO 综合判死循环 | `single_agent_deadloop` 检测器（连接频率 + 重复 prompt 哈希 + 文件重复读 + 低进度比） |
| **异常监测 · 基础（安全）** | 非预期 Shell、敏感文件越权、越界删除 | `unexpected_shell` / `sensitive_file_access` / `out_of_workspace_deletion` 三类检测器 |
| **异常监测 · 进阶** | 多智能体协作/竞争场景异常 | `cross_agent_prompt_injection`（CAPI 跨 Agent 提示词注入）、`mcp_tool_spoofing`（MCP 工具伪造，扩展项） |
| **告警与归因 · 基础** | 含时间戳、PID/TID、操作对象的日志 | 统一 `Alert` schema（`proto/alert.schema.json`），含 `ts_ns`、`pid`/`tid`、`objects.*`、`evidence[]` |
| **告警与归因 · 进阶** | 跨层数据关联，精确因果匹配 | `EvidenceChain` + CAPI 评分：上游工件 → 下游 Prompt → 高危 syscall，输出 `chain_id` 与因果 `explanation` |

### 1.3 与评审要点的对应

- **功能实现与异常回溯（40%）**：在 syscall 捕获与基础安全检测的基础上，实现了 Pixie/Falco/Tetragon 等通用工具均不具备的"Prompt → Syscall"跨层因果归因与多智能体根因定位（详见 §6）。
- **观测深度与兼容性（30%）**：tracepoint 精确获取进程/网络/真实文件路径上下文（路径规范化见 §4.2），uprobe 非侵入式截获 OpenSSL 明文；全程无需改内核或重编译目标程序。
- **系统性能与工程（30%）**：同负载、仅切换捕获模式的可复现基准下，核心数据捕获（syscall+TLS）开销约 +2%，严格满足 ≤5% 目标（详见 §8）。

## 2. 设计思路

### 2.1 第一性原则

1. **内核态轻采集、用户态重分析**：eBPF 程序只做最轻量的字段提取与 ringbuf 写入，**不做文本解析、JSON 解析或跨事件关联**；所有语义重活都在用户态完成。这是满足 ≤5% 性能预算的根本前提。
2. **分层闭环、逐层增强、独立可用**：系统按三层闭环组织，**每层独立可用**，上层失效时下层仍照常告警，差异只反映在 `fallback.mode` 上：

```text
基础层：  syscall            -> 规则检测器 -> 告警
增强层：  + TLS/stdio 明文   -> PromptObject -> prompt↔事件关联
深链层：  + SourceArtifact   -> 下游 Prompt -> 下游高危 syscall -> CAPI 告警
```

3. **诚实降级（Degradation Honesty）**：当 TLS 不可见、ringbuf 背压丢包或归因不完整时，系统**主动降级并显式标注** `visibility`/`fallback`，绝不把推测包装成强结论（例如降级时 CAPI 路径禁止输出 `critical`）。系统应"失之于浅，而非失之于盲"。
4. **先把一条跨层路径做可靠，再泛化**：主线只先打通**唯一一条**受控路径（工作区文件工件 → TLS Prompt → 高危 syscall），把这套机制做扎实后再向 stdio/pipe/MCP 等更多传播通道泛化。

### 2.2 关键设计取舍

- **只检测、不阻断**：定位为非侵入式"检测 + 归因"，不像 Tetragon 那样在内核态 kill/block。这是有意取舍，避免对智能体正常运行造成干扰风险。
- **多智能体一等建模**：`session` / `agent` / `role` 是一等实体。在进程交织的并发环境中通过 seeds + env + argv + connect 证据做 session election；`agent_id` 在角色升级时保持稳定，`lineage_id` 跨守护化（double-fork/setsid）churn 保持稳定。
- **传播门控先于文本相似度**：跨 session 关联必须先有**显式传播事实**（同一工作区文件血缘、管道/资源交接等），再考虑文本相似度——相似度只能给候选排序，不能凭空创造候选，从源头压制误报。
- **单写者所有权**：每个状态注册表只有唯一写者（如 `ProcessState` 仅由 normalizer 写、`AgentBinding` 仅由 graph 写、连接身份仅由 state-net 写），`NormalizedEvent` 归一化后不可变，下游通过按 `evt_id`/`ingest_seq` 索引的快照旁路 join，避免"越往后归因越好地反复覆写事件体"。

### 2.3 开发阶段规划

项目按 7 个阶段（Phase 0–6）演进，先稳定数据面，再建语义，再做跨层因果，最后硬化性能与扩展通道：

| 阶段 | 目标 | 状态 |
|---|---|---|
| Phase 0 骨架与最小可观测 | 加载、收事件、输出规范化告警的最小闭环 | ✅ |
| Phase 1 基础异常闭环 | 3 类基础安全检测 | ✅ |
| Phase 2 多智能体语义与资源滥用 | session/role 识别、并发隔离、死循环检测 | ✅ |
| Phase 3 TLS 提示词捕获与同会话关联 | OpenSSL uprobe 明文、同会话 prompt→高危关联 | ✅ |
| Phase 4 文件工件与跨会话 CAPI | 跨会话因果链 | ✅ |
| Phase 5 硬化：性能/降级/可复现 | 性能基准、降级诚实、IPC、复现脚本 | ✅ |
| Phase 6 扩展：更广传播通道 | content_io、MCP 解析/Tool Spoofing、IPC events/graph（扩展项） | ◐ 进行中 |

> 说明：赛题三大任务与三项评审要点均落在 Phase 0–5 且已全部达成并经实测验证；Phase 6 为纯扩展项，不阻塞赛题任务。详见 `docs/开发进度与完成情况.md`。

## 3. 总体架构与数据流

### 3.1 模块分层

```text
ingest（采集）
  BPF(内核) -> ringbuf -> collector -> 不可变 NormalizedEvent + 丢包遥测

state（状态）
  进程 / fd / cwd / 路径 / 挂载 / session / workspace 注册表
  + 独立的连接/套接字身份引擎(state-net)
  + 后台对账器(env 嗅探、启动期 /proc 合并、规范化缓存)

content（内容）
  全局 TLS 片段 -> 流重组 -> MessageBoundary -> PromptObject / SourceArtifact

decision（决策）
  派生信号 -> 检测器 -> 告警策略 -> sink(NDJSON、stderr、IPC)
```

数据交付契约（保证正确性与可复现性的关键不变量）：

- `collector -> normalizer` 保留每 CPU 事件顺序；normalizer 给每个外发事件打**单调 ingest 序号**。
- `NormalizedEvent` 在归一化后**不可变**，只携带解析与路径/进程归一化拥有的事实；session/agent 归因、连接身份、流所有权、内容派生对象都在之后通过旁路注册表发布。
- 所有对外输出（文件、stderr、IPC）都必须经过 `veriskein-alert` 与 `veriskein-ipc`；检测器**绝不**直接写 sink。

### 3.2 运行时实体模型

| 实体 | 来源 | 生命周期 |
|---|---|---|
| `Process` | BPF fork/exec/exit | 进程存活期 + 30s 过期保留（用于迟到归因） |
| `Session` | 确认的 root 候选及其继承的后代 | `root_pid` 退出后保留 5 分钟再驱逐 |
| `Agent` | session 内的角色分类 | 跟随进程 |
| `PromptObject` | 重组的内容片段 + 抽取 | 有界 LRU；窗口内保留完整摘录，之后仅 id/hash + 脱敏摘要 |
| `SourceArtifact` | 文件 / MCP / stdio / 上游响应 | 有界 LRU；压力下摘录可脱敏或降级为摘要 |
| `EvidenceChain` | correlator 构建 | 被至少一条告警引用才持久化 |
| `Alert` | 检测器输出经策略投影 | 追加写 NDJSON（`.jsonl`） |

## 4. 实现描述（按能力层）

### 4.1 多层级数据捕获（任务 1）

**BPF Hook 清单**（全部使用 CO-RE 字段访问，`vmlinux.h` 由 `bpftool btf dump` 生成）：

| 程序 | Hook | 类型 | 事件 |
|---|---|---|---|
| `proc.bpf.c` | `sched_process_fork/exec/exit`、`sys_enter_chdir/fchdir`、`sys_exit_dup*/close` | tracepoint | `EVT_PROC_*` / `EVT_FD_DUP` |
| `fs.bpf.c` | `sys_exit_openat/openat2/unlinkat/renameat2` | tracepoint | `EVT_FILE_OPEN/UNLINK/RENAME` |
| `net.bpf.c` | `sys_exit_connect`（`accept4` 可选） | tracepoint | `EVT_NET_CONNECT/ACCEPT` |
| `tls_uprobe.bpf.c` | `libssl.so:SSL_read{,_ex}` / `SSL_write{,_ex}` | uprobe+uretprobe | `EVT_CONTENT_FRAG(channel=tls)` |
| `content_io.bpf.c`（扩展项） | `sys_exit_read/write` | tracepoint | `EVT_CONTENT_FRAG(STDIO/PIPE/MCP)` |

实现要点：
- **优先 `sys_exit_*` 而非 kprobe**，以拿到返回值；需要入参时在 `sys_enter_*` 用 per-tid map 暂存、`sys_exit_*` 合并——避免把失败的 syscall 误判为成功的高危操作。
- 事件为 **little-endian、`__attribute__((packed))`、带版本号** 的二进制 ABI，统一头 `veriskein_evt_hdr`（含 `ts_ns`/`pid`/`tid`/`ppid`/`cgroup_id`/`mount_ns`/per-CPU `seq` 等）。任何字段变更必须 bump `EVT_ABI_VERSION`，用户态在入口拒绝 ABI 不匹配。
- **流身份在用户态构造**：内核只发原始身份输入（`(pid,fd,channel)` 或 `(pid,ssl_ctx,channel)`），用户态 `veriskein-state-net` 维护 `fd_version` 处理 fd 复用，`stream_id = hash(...)`，避免内核态维护易错的 fd-epoch 状态。

### 4.2 路径规范化（观测深度的关键）

路径层不依赖松散的布尔标志，而冻结一套共享类型 `PathResolution { lexical, canonical, mode, verdict, freshness_ns }`：

- `PathResolutionMode`：`LexicalOnly | Canonicalized | Unresolved`
- `PathVerdict`：`LexicalTrusted | CanonicalTrusted | CanonicalMismatch | UnresolvedSensitive`

热路径保持无 syscall：普通路径只做词法规范化（处理 `.`/`..`/冗余 `/`）；只有**敏感规则候选**与**工作区边界检查**才走 `readlink` 符号链接解析，且经缓存/对账器而非内联在事件路径上。检测器必须检查 `verdict` 而非盲目把 lexical 当 canonical——这既保证准确性又控制开销。

### 4.3 TLS 明文捕获（任务 1 · 进阶）

- 启动时 `veriskein-bpf` 从配置路径、动态链接器缓存与常见库路径**按序发现** `libssl.so.3` / `libssl.so.1.1`，按规范路径去重后**全局 attach** 一次（多个进程共享同一 DSO 时内核自动套用），而非按进程反复 attach。
- `SSL_read*/SSL_write*` 用 entry probe 暂存参数、return probe 用真实返回长度只发**成功的明文字节**。
- TLS 字节到达时进程归因可能还没完成：未归因片段按字节数、per-pid 队列长度、TTL 有界缓存，迟迟无法归属到 graph 确认/候选会话就在抽取前丢弃，并区分 captured/unattributed/dropped/promoted 指标。
- 明确不支持：静态链接 OpenSSL、BoringSSL、Go-native TLS、rustls、HTTP/2；TLS uprobe 目前仅 x86_64。这些通过 `unsupported` 可见性如实表达。

### 4.4 异常监测（任务 2）

五类核心检测器 + 一类扩展检测器。检测器**不直接产出 schema 告警**，而是产出内部 `Finding`（类型化证据 + 健康/可见性输入 + 组件分），由 `veriskein-alert` 统一映射 `severity`/`confidence`/`fallback`：

1. **`unexpected_shell`**：会话内进程 exec `{sh,bash,zsh,dash,fish}` 且不在 `shell_allowlist`；若前序 Prompt 含 shell 祈使语义则升级为 `critical` 并附 `prompt_id`。
2. **`sensitive_file_access`**：`open` 成功且路径（按 verdict）命中 `config/sensitive.toml`（如 `/etc/shadow`、`~/.ssh/id_*`、`~/.aws/credentials`）。
3. **`out_of_workspace_deletion`**：`unlink` 成功或 `rename` 目标落在工作区外，且不在 `delete_allowlist`（如 `/tmp/**`）。
4. **`single_agent_deadloop`**：120s 滑窗内命中**至少两条**子规则——连接频率、重复归一化 prompt 哈希、文件重复读、低进度比（`progress_signals / activity_attempts < 0.05`）。进度信号是显式且内容无关的（新建/修改工作区文件、测试/构建成功等）。
5. **`cross_agent_prompt_injection`（CAPI）**：见 §6。
6. **`mcp_tool_spoofing`（扩展项）**：JSON-RPC `method`/`tool_name` 与 `list_tools` 习得注册表不符、重复 `rpc_id`、结构异常等，叠加后续高危事件。

通用误报控制：会话过滤（丢弃非会话进程事件，避免普通用户 shell 误报）、上下文模板抑制、allowlist 优先、诚实降级。

### 4.5 告警与归因（任务 3）

统一告警 schema（`proto/alert.schema.json`，schema_version=1）包含归因定位所需的全部上下文：`ts_ns`、`pid`/`tid`、`session_id`/`agent_id`、`objects`（paths/ips/ports/prompt_ids/artifact_ids/event_ids/chain_id/…）、有序 `evidence[]`、`fallback`（降级/可见性）、`policy`（版本化、组件分）、`capture`（捕获模式与脱敏）。详见 §7。

## 5. 代码说明（按 crate / 目录）

工程为 Rust 2024 edition 的 Cargo workspace（`rust-toolchain.toml` 固定工具链），遵循团队代码组织约定：`lib.rs` 保持轻量、单测放 `src/tests.rs`、模块边界对齐所有权。

### 5.1 BPF 程序（`bpf/`）

|---|---|
| `bpf/vmlinux.h` | `bpftool btf dump` 生成的内核类型，供 CO-RE |

### 5.2 Rust crates（`crates/`，16 个）

| crate | 职责 | 不得做 |
|---|---|---|
| `veriskein-proto` | C 事件镜像（`#[repr(C, packed)]` + `plain::Plain`）、ID 类型、默认阈值、`parse()` 入口 | — |
| `veriskein-bpf` | libbpf 骨架绑定、全局 OpenSSL uprobe attach、DSO 发现/去重 | 文本/JSON 解析 |

| `veriskein-state-net` | 连接/套接字身份、端点历史、fd 复用版本、`ssl_ctx→connection` | prompt 解析、检测评分 |
| `veriskein-graph` | session 候选与选举、角色分类、per-session workspace 绑定 | 路径变更、流重组 |
| `veriskein-content` | 流重组、HTTP/1 与 JSON 边界、浅层 JSON-RPC 解析、Prompt/Artifact 抽取 | 评分、session election |
| `veriskein-correlator` | prompt/artifact/chain、MinHash、传播门控、CAPI 评分 | severity 决策 |

### 5.3 运行时主接线（`veriskein-daemon`）

`crates/veriskein-daemon/src/pipeline.rs` 的 `RuntimePipeline` 是 ingest→state→content→decision 的总装配点。关键路径（节选自 `process_owned_event`）：

```312:359:crates/veriskein-daemon/src/pipeline.rs
    fn process_owned_event(
        &mut self,
        ingest_seq: u64,
        event: &OwnedEvent,
        sink: &mut dyn Write,
        dry_run: bool,
    ) -> Result<Vec<AlertRecord>> {
        let mut emitted = Vec::new();
        for normalized in self.normalizer.apply(ingest_seq, event) {
            let graph_projection = self.apply_state(&normalized);
            self.ipc_events.push(ipc_event_projection(
                &normalized,
                self.graph
                    .resolve(normalized.process.pid)
                    .map(|binding| binding.session_id.hex()),
            ));
            if let Some(graph_projection) = graph_projection {
                self.ipc_graph.push(graph_projection);
            }
            self.drain_pending_prompts_for_event(&normalized);
            let prompt_evidence =
                prompt_evidence_for_event(&mut self.prompts, &self.graph, &normalized);
            let mcp_anomalies = self
                .pending_mcp_anomalies
                .remove(&normalized.process.pid)
                .unwrap_or_default();
            let mut findings = self.detectors.detect_with_prompt_and_mcp_evidence(
                &normalized,
                &self.graph,
                dry_run,
                &prompt_evidence,
                &mcp_anomalies,
            );
            self.observe_capi_file_event(&normalized);
            findings.extend(self.capi_findings_for_event(&normalized, &findings));
            for finding in findings {
                if let Some(alert) = emit_finding(
                    &mut self.alert_throttler,
                    sink,
                    &finding,
                    &self.runtime_health,
                )? {
                    emitted.push(alert);
                }
            }
        }
        Ok(emitted)
    }
```

设计要点：
- TLS/内容片段走单独的 `process_content_fragment` 路径产出 `PromptObject`；当 prompt 早于进程归因到达时，进入 `pending_prompts`（按 PID 暂存、有界、最旧优先驱逐），待归因补齐后回灌（`drain_pending_prompts_for_pid`）——解决了 TLS 字节先于会话确认到达的时序竞争。
- 内容捕获白名单随 fork 继承、随 dup/open/close/exit 维护（`observe_content_capture_policy`），保证 stdio/MCP 采样的 fd 语义正确。

## 6. 核心算法：跨层因果链与 CAPI 评分

这是项目最核心、也是赛题最看重（功能/回溯占 40%）的能力。

### 6.1 因果链结构

```text
工作区文件摘录(SourceArtifact) → 下游 Agent 的 TLS Prompt(PromptObject) → 下游高危 syscall
                       └──────────── EvidenceChain + CAPI 评分 ────────────┘
```

`PromptObject` 与 `SourceArtifact` 都保留稳定 ID、`hash_exact`、`hash_norm` 与 MinHash 签名；`EvidenceChain` 保留 `prompt_ids`/`artifact_ids`/`risky_event_ids`、`causal_score`、逐项 `component_scores` 与 `explanation`。

### 6.2 文本匹配（三级，取首个命中，不跨级叠加）

```text
normalize(bytes) = utf8_lossy -> NFKC -> lowercase -> 折叠空白 ->
                   去 markdown 围栏 -> 去引用前缀 "> " -> 截断 8192 字符
```

| 级别 | 条件 | 分值 |
|---|---|---|
| Exact | `hash_exact` 相等 | 0.40 |
| Normalized exact | `hash_norm` 相等 | 0.30 |
| Near-exact | MinHash Jaccard `j >= 阈值` | `0.20 * j` |
| Substring | 归一化连续子串 ≥ 64 字节 | 0.15 |

### 6.3 传播门控（Propagation Gating，误报防线）

文本相似度参与跨 session 候选**之前**，必须先满足至少一个具体传播事实（`PropagationFact`）：同一工作区文件血缘、显式管道/资源交接、或共享状态层记录的其它类型交接。**相似度只能排序候选，不能凭空创造候选。** 同时有上下文模板抑制：同一归一化哈希在 1 小时内出现于 ≥3 个 session 时按工作区/源类型分区抑制，但绝不隐藏含注入关键词或高熵的攻击载荷。

### 6.4 CAPI 评分

```text
score(chain) = match_score
  + session_bonus     (+0.15 跨 session 且存在显式传播通道)
  + window_bonus      (+0.15 高危事件在 prompt 后 30s 内)
  + risk_kind_bonus   (+0.10 shell | +0.10 sensitive | +0.05 connect)
  + injection_keyword (+0.10 归一化工件命中关键词)
  + weak_only_penalty (-0.20 仅子串匹配且无 exact/norm/near)
clamp 到 [0,1]
```

阈值：`>= 0.70` → `critical/strong`；`0.55~0.70` → `high/medium`；`< 0.55` 不出 CAPI（但证据可并入基础检测器）。**仅子串匹配的证据永远不得超过 high/medium。**

### 6.5 链构造硬约束

只有同时满足以下才构链：存在上游 session A 与下游 session B（`SingleAgent` 时 A==B）；存在 origin=A 的工件 X；传播门控通过且 `match_score>0`；B 在窗口内执行高/中危事件；时间顺序 `X.ts <= prompt_b.ts <= risky.ts` 一致。`explanation` 含一上一下两段脱敏摘录（各 ≤160 字节），API key/PEM/长 hex/base64/家目录身份在到达告警、IPC 或磁盘前必须脱敏。

## 7. 告警 Schema 与诚实降级

### 7.1 Schema 要点（`proto/alert.schema.json`）

必填字段：`schema_version`、`alert_id`、`ts_ns`、`type`、`severity`、`confidence_band`、`confidence_score`、`pid`、`tid`、`session_id`、`summary`、`reason_code`、`objects`、`evidence`、`fallback`、`policy`、`capture`。`additionalProperties:false` 严格约束。`alert_id` 由 `(session_id, type, 主对象)` 确定性派生，使重复检测器输出自动塌缩。

### 7.2 职责分离（防止设计退化的关键约束）

- 检测器只产 `Finding`（类型化证据 + 组件分 + `FindingHealth`），**不得**自行决定 `severity`/`confidence`/`fallback` 字符串。
- `veriskein-alert` 是**唯一**把 `Finding + 健康快照 → schema 合法告警`的层，独占 severity/confidence/fallback 的最终决定权与 schema 校验。
- sink 只做校验、节流、发布。

### 7.3 诚实降级语义

`fallback.mode ∈ {none, degraded, prompt_evidence_unavailable, capture_disabled}`，`fallback.visibility ∈ {full, partial, unsupported, unavailable}`：
- 10s 丢包率 > 阈值（1%）→ `degraded`：severity 降一级、band 封顶 `medium`。
- 应有 prompt 证据但 TLS/stdio 双双失败 → `prompt_evidence_unavailable`：**CAPI 禁止**，改发基础告警。
- 目标用不支持的 TLS 栈/静态链接 → `capture_disabled`：基础检测器不受影响，CAPI 禁止。

### 7.4 节流

key = `(session_id, type, reason_code, fallback.mode, 主对象哈希)`，60s 内去重（死循环检测器用自身 10 分钟冷却）；不跨 `fallback.mode`/`reason_code` 合并（它们代表实质不同的解释）；合并的 `event_ids` 有界，避免病态重复撑爆告警。

## 8. 测试、可复现与性能

### 8.1 三层测试

1. **单元测试**：`cargo test --workspace` 约 **190 个全部通过**，覆盖每个 crate 的核心不变量（ABI round-trip、路径边界、fd 复用、归一化幂等、MinHash 稳定、CAPI 分向量、schema 模糊等）。
2. **确定性回放（replay）**：把记录好的 NDJSON 事件流喂回用户态管线（不挂 BPF），用同一套断言 DSL 校验。**10/10 通过**：`alert_redaction`、`attribution_shell`、`benign_shell`、`cross_agent_prompt_injection(_parallel_streams)`、`deadloop_waste`、`out_of_workspace_deletion`、`sensitive_file_access_shadow`、`startup_shell`、`tls_prompt_same_session`。
3. **Live eBPF 场景**：`tests/run_all.sh` 真挂 BPF 跑端到端。**9/9 通过**：`benign_shell`(负例)、`unexpected_shell_basic`、`sensitive_file_access_shadow`、`out_of_workspace_deletion`、`deadloop_waste`、`tls_openssl_prompt_capture`、`cross_agent_prompt_injection_tls`、`cross_agent_prompt_injection_parallel_streams_tls`、`degradation_honesty`。

可复现性保障：外部主机名经本地 `/etc/hosts`/桩解析器（不依赖公网）；time 锚点用 `CLOCK_BOOTTIME` 对齐 `hdr.ts_ns`；每场景独立 `$SCRATCH`；敏感文件场景用 bind-mount 到 scratch + 临时规则而非动真格 `/etc`。

### 8.2 性能（评审要点：≤5% 数据捕获开销）

方法：**同一代表性负载、仅切换挂载的 BPF 程序**（用真实守护进程开关 `--disable-tls`/`--disable-content-io`/`--enable-content-capture` 隔离），交织取样 + 多趟中位数，公平对比"纯捕获开销"。`full` 模式让负载在 agent 包装下真实读写 stdio，真正命中内容捕获白名单（事件量约为纯内核模式 4–5 倍）。

实测（Linux 6.6 / WSL2，8 vCPU，release）：

| 模式 | 捕获内容 | 实测开销 | 门控 |
|---|---|---|---|
| kernel-only | 进程/文件/网络 syscall | **+2.18%** | ≤5%（严格）✅ |
| kernel+tls | + OpenSSL uprobe 明文 | **+2.27%** | ≤5%（严格）✅ |
| full | + stdio/MCP 内容采样 | **+4.04%** | 信息项 ✅ |

守护进程常驻内存 ~180–210 MiB。**诚实说明**：单次负载噪声约 ±6%，各百分比在重复运行间会浮动（kernel-only/kernel+tls ~0–2.3%，full ~4–6.5%）；稳健结论为"syscall+TLS 捕获损耗小且 ≤5%，内容捕获真实且受吞吐量约束"。持续大流量下全开模式可能逼近/超过 5%，生产环境以限速 + 按角色白名单约束，后续方向为内核态 cgroup/PID 预过滤。数据来源 `artifacts/perf-real/`。

## 9. 研发过程中遇到的问题和解决方法

> 以下问题与解决方法以 git 提交记录、`docs/ai-interactions/` 交互记录与实测为依据。

| # | 问题 | 解决方法 | 证据 |
|---|---|---|---|
| 1 | **失败 syscall 被误判为成功高危操作**（如 open 被拒仍当作越权访问，或反之漏报） | BPF 侧改用 `sys_exit_*` 保留返回值；同时即使 denied，也保留风险信号（攻击意图仍在），由检测器区分 | commit `fix: Preserve syscall results...`、`fix: Detect denied sensitive file access...` |
| 2 | **fork/exec 后 fd 与 cwd 状态丢失**导致路径上下文不准 | normalizer fd 表 fork 时 copy-on-write、exec 后保留未关闭 fd；启动期扫描 `/proc/*/{cwd,fd,comm,status}` 引导 | commit `refactor: Harden live event capture...` |
| 3 | **按进程反复 attach OpenSSL uprobe** 成本高且时序脆弱 | 改为按规范 DSO 路径**全局 attach 一次**，进程归因交给用户态 content/correlator | commit `refactor: Replace per-process TLS capture with global OpenSSL attachment` |
| 4 | **TLS 明文先于会话归因到达**（捕获在前、session 确认在后） | 未归因 prompt 进 `pending_prompts`（有界、最旧优先驱逐），归因补齐后回灌；未归因 TLS 字节按 TTL/队列/内存有界丢弃 | `pipeline.rs::drain_pending_prompts_for_pid`、单测 `pending_prompts_are_bounded_oldest_first` |

| 6 | **跨 session 文本相似度易产生误报** | 引入**传播门控**：必须先有显式传播事实再考虑相似度；加上下文模板抑制（但不掩盖注入关键词/高熵载荷） | 本文档 §6.3、`docs/ai-interactions/2026-06-21-22-...md` |
| 7 | **能力不可用时是否仍输出强结论** | 设计**诚实降级**：`VisibilityState` 统一映射 fallback，降级时不出 `critical`；新增 `degradation_honesty` 场景验证 | 场景 `degradation_honesty` |

| 11 | **`lib.rs` 易膨胀、单测堆积** | 遵循团队代码组织约定：crate root 轻量化、单测移入 `src/tests.rs`、按所有权边界拆分内部模块 | commit `refactor: Split detector, graph, alert, bpf, and collector crate roots...` |

## 12. 创新点、局限与后续工作

### 12.1 核心创新

1. **跨层语义因果链（Prompt → Syscall）**：现有工具按"层"切分（Pixie 看应用协议、Hubble 看网络、Falco/Tetragon 看 syscall），无一把"应用层模型输出"与"内核层系统行为"用因果关系连起来。本项目构建了受控、可解释、可评分的跨层因果链，直接回应赛题进阶要求与 40% 评审要点。
2. **多智能体一等建模与跨 Agent 归因（CAPI）**：session/agent/role 一等建模；Agent A 写入内容被 Agent B 读取并注入其 Prompt 进而触发高危操作时，用 MinHash/归一化签名 + **显式传播事实先于文本相似度**建立跨 session 因果链，输出 `chain_id`、`causal_score` 与多类证据。这是通用安全工具完全没有的能力维度。
3. **工程层面的诚实降级**：告警内置 `fallback.mode` 与 `visibility`，能力不可用时主动降级并显式标注，而非给出无依据的高置信结论。

### 12.2 局限（诚实声明）

1. **不做内核态阻断**：定位为非侵入式"检测 + 归因"，有意取舍。
2. **TLS 兼容性受限**：仅支持动态链接 OpenSSL（`libssl.so.1.1/3`），不支持静态链接/BoringSSL/Go-native TLS/rustls/HTTP/2；TLS uprobe 目前仅 x86_64。
3. **成熟度与形态**：面向赛题的研究型原型，单机、无可视化 UI（仅 daemon + CLI + NDJSON + IPC）。
4. **性能边界**：syscall 极端密集的突发场景下，全开内容捕获模式开销可能超过 5%（见 §8）。

### 12.3 后续工作（项目持续演进能力）

- 补 stdio/pipe 真实内容捕获 live 场景与独立 MCP Tool Spoofing live 场景（场景 F）。
- 落地内核态 cgroup/PID 预过滤 + 内容捕获自适应限速，进一步压低全开模式开销。
- 基于已预留且版本化的只读 IPC 协议接入只读可视化前端（如 Tauri）。
